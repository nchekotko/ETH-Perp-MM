# Роадмап тестирования market-making модели

Рабочий документ по уровням зрелости тестирования бэктестера (AS + funding-skew, ETH-perp, 3 дня данных).
Ссылки на код даны относительно корня репо. Текущая база: `pytest`, 7 файлов в `tests/`, ~40 тестов.

Обозначения: E — трудоёмкость (часы, оценка «сесть и написать»), КП — критерий прохождения.

**Цель:** каждый механизм движка и стратегии проверен изолированно; регрессии ловятся на уровне функции.

### 1.2 Дыры — что добавить

**(a) `_refresh_queue_positions` (`src/cmf_mm/engine/event_loop.py:56`) — сейчас не покрыт вообще.**
- инициализация: ордер на тач-уровне, `queue_ahead is None` → становится `book.bid_sz`/`ask_sz`;
- клэмп сверху: displayed size на нашем уровне уменьшился ниже оценки → `queue_ahead = min(старое, новое)`; увеличился → оценка НЕ растёт (монотонность);
- наша цена лучше тача (книга ушла) → `queue_ahead = 0.0`; глубже тача → остаётся `None`;
- переход уровня: книга ушла от нас (`queue_ahead=0`), затем вернулась — оценка остаётся 0 (мы «в голове»); зафиксировать это поведение тестом как осознанное допущение.

**(b) Матчер (`src/cmf_mm/engine/matcher.py:check_fills`) — дозакрыть queue-ветку.**
- переполнение очереди при двух ордерах на одном уровне: FIFO, overflow первого не даётся второму сверх `remaining = overflow − qty` (учёт строк 102–105);
- print-through при `queue_ahead > 0`: fill полного остатка независимо от очереди (есть частично, добавить кейс с partial: `trade.size < o.remaining`);
- консервация объёма: сумма fill-размеров по всем ордерам ≤ `trade.size` (queue: с учётом съеденной очереди);
- цена fill == цена лимитника, не цена принта (есть для touch, добавить для queue/cross);
- `partial_fills=False` в queue-модели (сейчас флаг игнорируется веткой queue — либо тест, фиксирующий это, либо баг-фикс).

**(c) Funding-начисление (`src/cmf_mm/engine/event_loop.py:171–179`) — частично покрыто `tests/test_funding.py` (знаки, величина, identity, realized-cash); дозакрыть остальное.**
- знаки: q>0, f>0 → `pay < 0` (лонг платит); q<0, f>0 → `pay > 0`; f<0 — зеркально;
- период: два FundingEvent с Δt=20 с, rate=r, mid=m, q=1 → `funding_pnl == −m·r·20/(8·3600)` точно;
- первый FundingEvent не начисляет (нет `last_funding_ts`);
- начисление идёт по **предыдущей** ставке (`last_funding_rate`), не по новой;
- funding при q=0 → 0; funding попадает и в `cash`, и в `funding_pnl` (одинаково).

**(d) Tick-округление и maker-clamp (`src/cmf_mm/strategy/as_funding.py:84–87`) — не покрыто.**
- bid округляется вниз, ask вверх (`floor`/`ceil` по `tick_size=0.1`), значения кратны тику с точностью 1e-9;
- clamp: bid ≤ `book.ask_px − tick`, ask ≥ `book.bid_px + tick` (никогда не пересекаем противоположный тач при агрессивном skew);
- при спреде книги ровно 1 тик котировки не самопересекаются: `bid_quote < ask_quote`;
- funding-skew: f>0 → обе котировки ниже, чем при f=0 (знак `−funding_kappa·f·s`); f<0 — выше; `funding_kappa=0` → эквивалент базовой AS вокруг микропрайса;
- односторонность у лимита: `q == max_inventory` → только sell; `q == −max_inventory` → только buy.

**(e) Инвариант декомпозиции с funding.**
- `tests/test_metrics.py::test_pnl_decomposition_synthetic_full_path` расширить веткой с funding: ручная последовательность fill → mid move → funding transfer, проверка `spread+inventory+funding == cash + q·mid` в лоб;
- `tests/test_engine_integration.py` — добавить FundingEvent в `_events()` и `ASFundingStrategy`, инвариант проверяется автоматически (`PnLDecomposition.__post_init__` кидает AssertionError);
- комиссии: `fee_bps=1.0` → fee вычитается и из `cash`, и из `spread_capture` (строки 160–163 event_loop), инвариант держится.

**КП:** все новые тесты зелёные; покрытие `engine/` и `strategy/as_funding.py` ≥ 90% строк (`pytest --cov`).
**E:** 6–8 ч.

## Уровень 2. Property-based тесты (hypothesis)

**Цель:** инварианты держатся не на подобранных примерах, а на произвольных потоках событий; фаззер находит краевые случаи (пустая книга, нулевые размеры, ставки-выбросы).

Добавить `hypothesis>=6.100` в `[project.optional-dependencies].dev` (`pyproject.toml`). Новый файл `tests/test_properties.py`.

**Стратегии генерации:** последовательности `LOBEvent | TradeEvent | FundingEvent` с валидными книгами (bid < ask, размеры ≥ 0, спред 1–5 тиков), ценами около 100 ± несколько тиков, ставками f ∈ [−0.01, 0.01], размерами трейдов из log-uniform.

**Свойства:**

1. **Identity-фаззинг:** прогнать `run_backtest` c `ASFundingStrategy` на произвольном потоке (50–500 событий) — конструкция `BacktestSummary` не кидает `AssertionError` из `PnLDecomposition`; дополнительно `cash + q·mid == spread+inv+funding` пересчитать снаружи с tol из движка. Ключевой тест уровня — он единственный проверяет взаимодействие fee × funding × partial fills × реквоты сразу.
2. **Нет самопересечения котировок:** для произвольных `(book, q, f, imbalance, gamma, funding_kappa, alpha)` — `on_event` даёт `bid < ask`, обе цены кратны тику, bid ≤ ask_px−tick, ask ≥ bid_px+tick.
3. **Монотонность `queue_ahead`:** для одного ордера на фиксированной цене под произвольным потоком LOB/Trade: `queue_ahead` не возрастает, пока цена — тач; никогда не отрицателен; ≤ displayed size тач-уровня в момент клэмпа.
4. **Консервация матчера:** для произвольного трейда против произвольного набора ордеров: Σ fill.size ≤ trade.size; `o.filled ≤ o.size`; fills не пустые только при правильной стороне агрессора.
5. **Доминирование моделей (по-трейдово):** на одном и том же (trade, orders) суммарный fill-объём: `cross ≤ queue ≤ touch` (queue с любым `queue_ahead ≥ 0`).

**КП:** 200+ примеров на свойство без падений; найденные контрпримеры разобраны (либо баг-фикс, либо задокументированное допущение с сужением стратегии генерации).
**E:** 6–10 ч (половина уйдёт на стратегии генерации валидных книг).

## Уровень 3. Интеграционные тесты с аналитическим PnL

**Цель:** движок целиком (loop + matcher + OM + декомпозиция) воспроизводит PnL, посчитанный на бумаге. Новый файл `tests/test_engine_analytic.py`. Удобно завести стуб-стратегию `FixedQuoteStrategy` (котирует заданные цены, игнорирует state) — 15 строк.

**Сценарии:**

1. **Round-trip без движения mid:** книга 99.9/100.1; котировки 99.9/100.1; sell-агрессор 99.9 → buy fill; buy-агрессор 100.1 → sell fill. Ожидание: `total == 2·0.1·size`, всё в `spread_capture`, `inventory_pnl == 0`, `funding_pnl == 0`, конечный q=0.
2. **Детерминированный проход через уровень:** пять трейдов sell-агрессора по убывающим ценам через наш bid c `queue_ahead=Q` — вручную посчитать, какой трейд начинает наполнять нас (после поглощения Q), сколько partial fills, итоговый `filled`. Сравнить fills от движка 1-в-1 (ts, size, price).
3. **Чистый funding без филлов:** один buy-fill размера q (получить инвентарь), затем только FundingEvents с постоянной f и неподвижным mid m на горизонте T. Ожидание: `funding_pnl ≈ −q·m·f·T/T_f` (T_f = 8 ч), `spread_capture` — только от fill, `inventory_pnl == 0`; знак меняется при f → −f и q → −q.
4. **Чистый inventory drift:** fill размера q, затем LOB-события двигают mid на Δ без трейдов → `inventory_pnl == q·Δ` точно, `total == spread_at_fill + q·Δ`.
5. **Комиссии:** сценарий 1 с `fee_bps=b` → `total == 2·0.1·size − b·1e-4·(оборот)`; при b, равном half-spread в bps, PnL раунд-трипа ≈ 0 (break-even sanity).

**КП:** равенство с аналитикой до `1e-9` (сценарии 1, 2, 4, 5) и `1e-6` (3, из-за дискретизации 20-с интервалов).
**E:** 4–6 ч.

## Уровень 4. Sensitivity / робастность

**Цель:** результат не является артефактом одной точки параметров или оптимистичной fill-модели. Инфраструктура уже есть: CLI-оверрайды `scripts/run_takehome.py` (`--gamma --funding-kappa --order-size --fill-model --fee-bps --tag`) и `scripts/run_sweep.py` (joblib-сетка по dotted-ключам YAML). Оформить как `scripts/run_sensitivity.py` или Makefile-цели, результаты — таблицей в отчёт.

**Кейсы:**

1. **Сетка параметров:** gamma × funding_kappa × order_size (например 4×4×3, walk-forward все 3 дня). Смотреть: total_pnl OOS, inv_std, max_drawdown, fills_per_hour. КП: PnL-поверхность гладкая (нет изолированного пика в одной ячейке — признак оверфита); выбранная точка не на краю сетки; знак OOS PnL сохраняется в окрестности ±1 шаг сетки.
2. **Три fill-модели как границы:** прогнать идентичный конфиг с `--fill-model cross|queue|touch`. КП: по каждому дню `spread_capture(cross) ≤ spread_capture(queue) ≤ spread_capture(touch)` и то же по fills_per_hour; для total_pnl трактовать как эвристику (инвентарные траектории расходятся — допустимы малые нарушения, но большие требуют разбора). Вывод в отчёт: «истина между cross и touch, queue — точечная оценка».
3. **Комиссии 0 / 0.5 / 1 / 2 bps:** КП: PnL монотонно убывает по fee; найти break-even fee и сравнить с реальными maker-фи (у топ-бирж maker ≈ 0–2 bps, часто рибейт) — если break-even < 1 bps, стратегия нежизнеспособна, писать честно.
4. **Латентность:** сдвиг реакции стратегии на 50/100/250 мс. Реализация: буфер отложенных `reconcile` в `run_backtest` (параметр `action_latency_us` в `BacktestConfig`, ~20 строк) — действия применяются при первом событии с `ts ≥ decision_ts + latency`. КП: PnL деградирует плавно, не обнуляется при 50 мс (иначе стратегия живёт на нереализуемой скорости реквот); зафиксировать таблицу PnL(latency).

**E:** 6–8 ч (латентный шим — половина).

## Уровень 5. Валидация симулятора против рынка

**Цель:** убедиться, что fill-модель порождает реалистичную частоту и качество исполнений, а не фантазию симулятора.

1. **Частота филлов vs λ(δ):** из `summary.fills` и записанной серии котировок собрать: время, проведённое котировкой на расстоянии δ от mid (бакеты по тикам), и число филлов в каждом бакете → эмпирическая интенсивность исполнения. Сравнить с калиброванной `A·exp(−k·δ)` (`src/cmf_mm/calibration/intensity.py`). КП: эмпирическая кривая лежит **ниже** λ(δ) (мы не забираем больше потока, чем есть на рынке — необходимое условие) и в пределах фактора ~2 по форме; сильное превышение = слишком оптимистичный матчер.
2. **A/B fill-моделей по микрометрикам:** fills_per_hour, avg_fill_size, avg_edge_usd, доля partial fills (`metrics.json` per-day, есть в `day_metrics`). КП: queue даёт fills_per_hour в разы ниже touch (при спреде почти всегда 1 тик очередь — главный ограничитель); touch при ~29k трейдов/день не должен давать fill-rate, близкий к 1 на каждый трейд у тача.
3. **Adverse selection / edge:** гистограмма edge уже есть (`plot_fill_edge_hist`, `src/cmf_mm/reports/takehome.py`). Добавить markout: `mid(t+h) − fill_price` (со знаком стороны) для h = 1 с / 5 с / 30 с. КП: средний markout не съедает больше ~50–70% сырого edge (иначе весь «спред-захват» — это adverse selection, и queue-модель просто маскирует его); распределение edge не должно быть двугорбым с массой в глубоко отрицательной зоне.
4. **Sanity объёма:** наш исполненный объём как доля рыночного (turnover_quote vs объём трейдов за день). КП: < 1–2% — иначе симулятор игнорирует собственное влияние на рынок.

**E:** 8–12 ч (markout-скрипт + скрипт эмпирической интенсивности, оба поверх сохранённых `summary_*.pkl`).

## Уровень 6. Статистическая значимость

**Цель:** честно оценить, что можно и что нельзя утверждать по 3 дням (из них 2 OOS-дня).

1. **Прямо сказать в отчёте:** 3 дневных PnL — это n=3 (OOS n=2); никакой t-тест/бутстрэп по дням не даёт значимости. Дневной бутстрэп не делать вовсе (или показать CI шириной «от −∞ до +∞» как иллюстрацию).
2. **Блочный бутстрэп внутри дня:** минутная сетка PnL уже есть (`minute_grid_pnl` в `scripts/run_takehome.py`) → приросты по 1 мин → блоки 30–60 мин (сохраняем автокорреляцию) → stationary/circular block bootstrap (1000 ресэмплов) → CI дневного PnL и Sharpe. КП: сам факт наличия CI в отчёте; если CI дневного PnL включает 0 (почти наверняка включает) — так и написать.
3. **Консистентность вместо значимости:** аргументировать через согласованность, а не p-value: (а) знак PnL и структура декомпозиции (spread_capture > 0, funding_pnl согласован со знаком средней ставки) одинаковы во все дни; (б) результат устойчив по сетке уровня 4; (в) fill-модель консервативна (уровень 5). Это единственная честная линия защиты на 3 днях.
4. **Deflated Sharpe / поправка на подгонку:** если перебиралось K конфигураций до выбора финальной — сообщить K и показать, что финальная точка выбрана по дню 1 (in-sample), а дни 2–3 не участвовали в выборе.

**E:** 4–6 ч (бутстрэп-скрипт поверх `summary_*.pkl` + честный абзац в отчёт).

## Уровень 7. Продвинутое

**Цель:** масштабирование протокола за пределы тейк-хоума; понимание, когда модель ломается.

1. **Multi-day sweep с walk-forward:** объединить `run_sweep.py` (сетка) с walk-forward-протоколом `run_takehome.py` (сейчас sweep гоняет `run_from_config` без walk-forward) — каждая ячейка сетки = полный 3-дневный walk-forward; отчёт по OOS-суммам. E: 4 ч.
2. **Стресс-дни:** синтетика на базе реальных данных — масштабировать лог-доходности mid ×2/×3 (пересобрать LOB parquet со «взорванным» mid), проредить/утолщить поток трейдов. КП: инвентарные лимиты держат `inv_max_abs`; PnL деградирует за счёт inventory_pnl, а не через нарушение инварианта/NaN; при σ×3 стратегия расширяет спред (проверить через средний quoted spread). E: 6–8 ч.
3. **Стабильность калибровки:** данные уже пишутся в `metrics.json → calibration` (σ_usd, A, k, R² по дням). Добавить проверки: R² ≥ 0.9 на каждом дне (иначе λ-фит мусорный — сейчас нигде не гейтится, только логируется); дрейф |Δk|/k и |Δσ|/σ день-к-дню < ~30–50%; если дрейф выше — walk-forward на вчерашних параметрах систематически котирует не тот спред, показать это сравнением PnL «вчерашние vs сегодняшние (оракульные) параметры». E: 3–4 ч.
4. **Режимные фильтры:** на базе `scripts/regime_check.py` — детектор режимов (realized vol в скользящем окне 5 мин vs калиброванная σ; funding-выбросы). Правило: при vol > 2×σ_calib — расширять спред или снимать котировки; A/B прогон с фильтром/без. КП: фильтр уменьшает худший внутридневной drawdown, не убивая > 30% PnL спокойных периодов. E: 8–12 ч.
